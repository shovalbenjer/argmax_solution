Refined Project Plan: A Soundproof Dietary Classifier

Project Goal: To engineer a highly accurate and robust dietary classifier. The system will be evaluated against a nuanced, LLM-generated ground truth dataset.

Core Architecture: The Hybrid Retrieval Cascade

This architecture is designed for speed, accuracy, and transparency. It avoids the pitfalls of simple keyword matching and pure LLM systems by combining three layers of analysis before making a judgment.

Symbolic Search (SQL): An instant lookup in our local knowledge_graph.db for known ingredients. This is the fastest path.

Syntactic Search (rapidfuzz): A fuzzy-matching layer to handle typos and minor variations (e.g., "flour" vs. "floud"), making the system robust to data noise.

Semantic Search (Embeddings): A "soundproof" layer that understands the meaning of words to detect contradictions (e.g., "egg" vs. "eggless"). This prevents critical misclassifications.

The output of this cascade is a rich structured_context object that is then fed to the Qwen LLM for a final, informed decision.

### **Refined Project Plan (V3): Unit-Aware, Text-to-SQL-Augmented Classification**

**Project Vision:** (Unchanged) To build a highly accurate and transparent dietary classification system where a "student" LLM (Qwen) learns from a "teacher" LLM (Gemini). The core innovation is a Text-to-SQL context engine (**RSL-SQL**) that provides rich, structured data from a local knowledge graph to both LLMs, enabling nuanced, explainable, and auditable judgments.

**Core Architecture (V3): The Full-Spectrum Analysis Cascade**

This is the definitive architecture that handles raw ingredient data end-to-end.

1.  **Ingredient Parsing:** A raw ingredient string (e.g., `'3 cups all-purpose flour'`) is first processed by `ingredient-parser-nlp` to break it into its core components: amount (`3`), unit (`cups`), and name (`'all-purpose flour'`).
2.  **Question Formulation:** A detailed natural language question is constructed, incorporating the parsed components. This question guides the Text-to-SQL model to perform normalization.
3.  **Text-to-SQL (RSL-SQL):** The question is sent to RSL-SQL [[1]](https://github.com/Laqcce-cao/RSL-SQL). The model uses its understanding of the database schema (including the `unit_conversions` table) to generate a complex SQL query that joins tables and calculates the final nutritional values.
4.  **Knowledge Graph Query & Normalization:** The generated SQL is executed against the `knowledge_graph.db`. This query will, for example, look up `'all-purpose flour'`, find its grams-per-cup conversion factor, and calculate the total carbohydrates for `3` cups.
5.  **LLM Judgment (Teacher/Student):** The final, normalized data (e.g., "136g total carbs") and vegan context are fed into the main LLM with a chain-of-thought prompt for a highly informed final verdict.
